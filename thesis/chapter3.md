# 第3章 数据采集驱动的知识构建链路实现（OpenClaw Skill 到结构化事件）

## 3.1 引言

在本研究中，第3章承担“方法可落地性证明”的核心角色。若第2章回答了“系统为什么这样设计”，那么第3章必须回答“系统如何在真实数据条件下稳定运行”。与理想化实验不同，足球新闻知识库面向的是持续更新、来源异构、表达噪声较高的动态文本流。尤其在新闻采集环节，数据质量并不由模型控制，而是由采集策略、过滤规则、入库约束和字段标准化共同决定。换言之，后续事件抽取、真实性验证和 GraphRAG 问答性能的上限，首先由本章讨论的数据工程链路决定。

本项目采用 OpenClaw 的 Skill 机制组织采集能力，具体体现为 `datasource/SKILL.md` 所定义的“任务目标 + 执行约束 + 质量标准”三层规范。该 Skill 并非简单脚本说明，而是可执行的采集协议：它规定了数据源优先级、技术栈边界、过滤标准、入库验收口径以及异常场景的务实处理方案。在工程实践中，这种“策略即文档、文档即约束”的方式能显著提升团队协作一致性，也能让自动代理在多轮执行中保持稳定行为。

本章围绕“从 OpenClaw Skill 到 PostgreSQL，再到预处理与抽取链路”展开，核心目标有三点：

1. 说明数据来源为何选择英超官网并采用分栏目优先策略；
2. 说明采集、清洗、分类、入库如何形成可复现的数据质量闭环；
3. 说明数据库记录如何通过契约化适配进入预处理与事件抽取流程。

与第4章和第5章相比，本章强调“输入端真实世界约束”，即反爬机制、栏目分布不均、非报道噪声比例、字段缺失与格式不一致等问题。只有在这一层构建出稳健的数据基础，后续的置信传播与检索问答才能具备可信性与可解释性。

## 3.2 数据来源与采集策略设计

### 3.2.1 选择英超官网作为主数据源的原因

足球新闻可从媒体聚合站、俱乐部官网、社交平台和第三方 API 获得，但本项目在研究阶段优先使用英超官网（Premier League News）作为主数据源，理由如下：

1. **主题一致性高**：英超官网新闻与研究任务（英超事件理解）目标高度一致，降低跨联赛、跨语言噪声。
2. **事件密度高**：比赛报道（Match Reports）天然包含比分、进球、阵容、教练发言、赛后评价等多种事件要素，适合事件抽取。
3. **时间连续性好**：联赛周期内新闻更新稳定，便于进行增量采集与时间序列验证。
4. **来源可标注性强**：官方域名数据可天然赋予较高来源可信先验，为第4章验证模型提供先验基础。

在毕业设计场景中，数据源“可解释、可复用、可积累”比“绝对规模最大”更重要。英超官网虽然存在反爬限制，但其内容结构更契合事件知识构建任务，适合构建可分析的实验数据集。

### 3.2.2 OpenClaw Skill 的采集目标与约束

`SKILL.md` 将采集任务明确为“从英超官网获取有效新闻报道并落库到 PostgreSQL `news` 表”。其中最关键的三类约束如下。

#### （1）强制技术约束

- 必须使用 Playwright 作为主要抓取手段；
- 原因是目标站点存在动态加载，纯 `httpx + BeautifulSoup` 难以稳定获取完整列表与正文；
- 对 AI 代理而言，这一约束降低了“错误工具选择”导致的失败率。

#### （2）数量验收约束

- 用户提出“爬取 50 篇”时，含义为“最终新增落库 50 篇”；
- 不是尝试 50 个链接，更不是抓到 50 条原始页面；
- 因此执行策略必须考虑过滤率、重复率和解析失败率，动态提高抓取尝试量。

#### （3）质量过滤约束

- 必须过滤互动投票、测验、视频集锦、纯榜单、广告推广、播客等非报道内容；
- 报道文本长度过短或正文提取失败也应剔除；
- 仅保留具备“事实信息承载能力”的新闻文本。

这三类约束共同定义了采集层的工程逻辑：不是“能爬就行”，而是“以最终可用数据为导向”。

### 3.2.3 栏目优先级机制

Skill 中将栏目按业务价值划分为 P0~P3：

- **P0：Match Reports**（最高优先）
- **P1：From The Clubs**
- **P2：Transfers**
- **P3：Latest News（默认）**

该策略具有明确的信息工程意义：

1. **提高事件抽取收益**：P0 报道通常包含明确时间、参与者、比赛结果和动作约束，结构化价值最高；
2. **降低噪声输入占比**：P3 综合栏目混入更多泛内容，直接全量抓取会稀释有效样本密度；
3. **支持资源受限场景**：在反爬和执行时间受限时，先抓高价值栏目可获得更高单位成本收益。

从知识图谱角度看，栏目优先级本质上是上游“信息价值排序”，它会直接影响下游图谱中事件节点的质量分布。

## 3.3 采集执行架构：OpenClaw Skill 到数据库

### 3.3.1 Skill 驱动的执行流程

结合 `SKILL.md` 与 `datasource` 模块，数据采集执行链路可抽象为六步：

1. **任务解析**：读取用户目标（如数量、过滤要求）；
2. **栏目调度**：按 P0→P1→P2 顺序分配抓取预算；
3. **页面抓取**：通过 Playwright 获取新闻列表与详情页；
4. **正文提取与清洗**：剔除导航、交互文本、模板噪声；
5. **有效性判断与来源类型分类**：筛除非报道并标注 `source_type`；
6. **写入 PostgreSQL**：按 URL 去重，记录新增与跳过统计。

该流程体现了“先质量控制，再持久化”的原则。即使采集到较多页面，只有通过质量门禁的数据才进入数据库，保证下游处理链路输入可用。

### 3.3.2 可行性与反爬现实约束

Skill 文档明确记录了实际环境中的反爬限制：

- 目标站可能返回 Access Denied；
- 栏目页面可见链接数量有限；
- 即便 Playwright 可访问，也可能单轮仅获得少量有效新闻。

因此系统采用“务实累积”策略：

1. 单轮尽可能获取高质量数据；
2. 多轮定时任务积累总量；
3. 对无法一次达标的任务给出透明说明与替代方案。

这与论文实验设计高度一致：在真实约束下维持可持续数据流，比追求一次性理想数量更符合工程实践。

### 3.3.3 异常与重试策略

在执行层面，采集任务主要面临以下异常：

- 页面加载超时；
- DOM 结构变化导致解析失败；
- 正文抽取为空；
- 数据库连接暂时不可用；
- 唯一键冲突（重复 URL）。

可行的稳健策略包括：

1. **请求级重试**：列表页与详情页采用有限次重试；
2. **任务级降级**：某栏目失败时继续后续栏目，避免全任务中断；
3. **入库级幂等**：依赖 `url UNIQUE` 与 `ON CONFLICT DO NOTHING`；
4. **结果级审计**：记录“新增/过滤/重复/失败”计数，支持复盘。

这些策略并不追求“零失败”，而是追求“失败可控、结果可解释、流程可继续”。

## 3.4 数据清洗与报道有效性判定

### 3.4.1 为什么要严格过滤非报道内容

如果不做清洗，新闻站点中的互动内容（投票、测验、视频推荐）会被当作普通文本进入抽取流程，造成三类问题：

1. **语义噪声升高**：事件抽取会产生大量无意义或伪事件；
2. **图谱污染**：无业务价值的“排行榜描述”被写入事件节点；
3. **问答偏差**：RAG 检索到非事实内容，降低答案可信度。

因此，数据清洗不是附属步骤，而是知识质量控制的第一道防线。

### 3.4.2 规则型过滤机制

Skill 中给出了明确过滤类别，可归纳为规则型判定集合：

- 标题关键词过滤：`rank`、`vote`、`quiz`、`watch`、`podcast` 等；
- 内容模式过滤：`your favourite`、`rank them`、投票式语句；
- 长度阈值过滤：正文短于最小长度阈值（如 100~200 字）判为低价值；
- 结构完整性过滤：无正文主体或仅短句列表判为无效。

规则过滤的优势是可解释、可快速迭代。其劣势是对边界样本敏感，可能存在误杀或漏判。因此实践中可结合抽样复核持续优化规则词表。

### 3.4.3 来源类型分类策略（source_type）

为支撑第4章验证模块的来源先验，采集层需要尽早标注 `source_type`。Skill 提供了基于关键词和结构特征的启发式分类：

- **OFFICIAL**：官方声明、确认、签约、任命等；
- **MEDIA**：分析、评论、赛前预览、赛后综述；
- **USER_GENERATED**：用户投稿、社区互动（该源中较少）；
- **UNKNOWN**：无法确定。

这一分类虽为启发式，但对验证环节十分关键：

1. 在来源加权阶段提供先验标签；
2. 为冲突事件解释提供证据（官方 vs 媒体）；
3. 为后续统计分析提供维度（不同来源类型的事件质量差异）。

### 3.4.4 清洗与分类的误差控制

为了降低误差影响，可采用以下控制策略：

- **双阈值策略**：对可疑样本设“人工复核池”；
- **日志采样复查**：每批次抽样检查过滤结果；
- **规则版本化**：记录关键词词表版本，便于比较改动影响；
- **下游反馈闭环**：若抽取层持续报错某类样本，回溯清洗规则修订。

通过这种闭环机制，采集质量可以随系统运行逐步提升，而非一次性固定。

## 3.5 PostgreSQL 存储模型与入库策略

### 3.5.1 `news` 表结构设计

本研究的原始新闻持久化采用 `news` 表，核心字段包括：

- `id`：主键；
- `url`：唯一标识（去重依据）；
- `title`：标题；
- `content`：正文；
- `publish_date`：发布日期；
- `source_name`：来源名称（默认 Premier League）；
- `source_type`：来源类型；
- `author`：作者；
- `crawled_at`：抓取时间；
- `created_time`、`updated_time`：记录生命周期字段。

同时建立 `url`、`publish_date`、`source_type`、`crawled_at` 索引，为后续增量读取和统计分析提供支撑。

### 3.5.2 幂等写入与重复处理

在重复抓取不可避免的场景下，系统采用 URL 唯一约束实现幂等：

- 写入语句使用 `ON CONFLICT (url) DO NOTHING`；
- 新增数量由 `cursor.rowcount` 或返回值判断；
- 批次结果拆分为“新增/跳过/失败”三类统计。

这种策略使任务可安全重跑，不会因重复数据导致数据库膨胀，也便于实现定时采集。

### 3.5.3 日期字段标准化问题

新闻发布日期可能出现多种格式（`YYYY-MM-DD`、`YYYY/MM`、`YYYY` 等）。项目在 `news_repository.normalize_publish_date` 中执行归一化，目标格式为 `YYYY-MM-DD`。若无法解析则允许回退到 `crawled_at` 日期。

这一处理能显著提升后续时间过滤稳定性，避免“日期字符串不可比较”导致的检索偏差。

### 3.5.4 入库验收口径

Skill 的验收核心并非“爬了多少 URL”，而是“最终新增落库数量”。因此采集报告至少应包含：

1. 用户目标数量；
2. 实际尝试抓取数量；
3. 过滤掉的非报道数量；
4. 已存在而跳过的数量；
5. 最终新增入库数量。

该口径让任务结果可量化、可对账，也与数据工程中的 SLA 思维一致。

## 3.6 Datasource 模块实现与契约化适配

### 3.6.1 仓储层：`PostgresNewsRepository`

`datasource/news_repository.py` 提供了三类关键读取接口：

1. `list_news`：支持分页、来源过滤、时间过滤；
2. `list_news_incremental`：基于 `(crawled_at, id)` 游标增量读取；
3. `get_news_by_ids`：用于定点样本复现（如 `id=10`）。

其中增量读取采用“时间优先、ID 次序”策略，能处理同一时间戳下多条记录的稳定遍历问题。这一设计可避免漏读和重复读取，是持续更新管道的基础。

### 3.6.2 适配层：`preprocess_adapter`

`datasource/preprocess_adapter.py` 将数据库记录转换为下游可消费的 `PreprocessDocument`，关键映射包括：

- `doc_id = news-{id}`；
- `raw_text = content`；
- `publish_date` 若缺失则回退 `crawled_at`；
- `metadata` 保留 `created_time` 与 `updated_time`。

通过适配层隔离了“数据库字段命名”与“算法模块契约”，后续即便数据库结构变化，也可以在适配层局部修复而不破坏主链路。

### 3.6.3 服务层：`DataSourceService`

`datasource/service.py` 面向上层流程提供三个方法：

- `fetch_documents_for_preprocess`（批量）；
- `fetch_incremental_documents`（增量）；
- `fetch_documents_by_ids`（定点）。

这意味着预处理模块无需感知 SQL 或数据库细节，只处理统一文档对象，系统耦合度显著降低。

### 3.6.4 契约稳定性的工程价值

在多模块流水线中，契约稳定性的作用主要体现在：

1. **可测试**：可用固定 `doc_id` 执行端到端回放；
2. **可维护**：上游改动不必同步改动所有下游模块；
3. **可审计**：每个事件可回溯到原始新闻记录。

这也是本项目能够持续迭代第4章与第5章能力而不频繁破坏主链路的关键原因。

## 3.7 从新闻文本到事件对象：预处理与抽取链路强化

### 3.7.1 预处理阶段：分句与语义分块

数据库文档进入预处理后，先经过句子切分，再经语义分块。语义分块采用“LLM 连续评分 + 阈值决策”机制：当相邻句语义断裂分数超过阈值时触发新块，并结合最大句数约束防止块过长。

该阶段输出 `SemanticChunkDocument`，包含 `block_id`、`block_text`、来源信息与时间信息。其核心目标是将长文本转化为更稳定的抽取单元。

### 3.7.2 抽取阶段：事件分解与锚点抽取

抽取分为两层：

1. **事件分解层**：一个语义块拆分为 1-N 个事件单元，并规范化 `event_id` 为 `{chunk_id}:e{序号}`；
2. **锚点抽取层**：在事件单元上抽取参与者、约束、时间锚点、来源等结构化字段。

此两阶段设计避免了单次模型调用任务过载问题，提高了结果结构稳定性和可解释性。

### 3.7.3 与第4章验证模块的接口耦合点

第4章需要的核心输入包括 `sources`、`participants`、`constraints`、`temporal_anchors` 与事件描述。上述字段在第3章抽取链路中已形成标准输出，因而可直接进入多源验证模块。

由此可见，第3章不仅是“数据准备”，更是第4章可信度建模的前提层。

## 3.8 数据质量评估与运行指标设计

### 3.8.1 采集质量指标

为量化采集有效性，建议在每轮任务记录以下指标：

- `attempted_count`：尝试抓取条目数；
- `valid_news_count`：通过过滤的有效报道数；
- `inserted_count`：新增入库数；
- `duplicate_count`：重复跳过数；
- `filtered_count`：非报道过滤数；
- `failed_count`：抓取或解析失败数。

这些指标可用于评估策略调整效果，如栏目配比、过滤词表变更、重试参数优化等。

### 3.8.2 文本质量指标

在进入预处理前，可进一步记录：

- 正文字数分布（均值、分位数）；
- 空正文比例；
- 日期缺失比例；
- 作者缺失比例；
- 不同 `source_type` 样本占比。

这类统计有助于识别结构性偏差，例如某栏目虽抓取量高，但正文过短比例高，则应降低其调度权重。

### 3.8.3 链路稳定性指标

在端到端层面，可增加：

- 每篇文档分句数、分块数；
- 每块平均事件数；
- 抽取失败率；
- 验证前后事件可用率。

通过链路指标可定位质量瓶颈发生在“采集端、预处理端还是抽取端”，从而实现针对性优化。

## 3.9 工程化落地与可持续迭代策略

### 3.9.1 定时采集与增量处理

考虑到目标站更新节奏和反爬现实，系统适合采用“定时小批量 + 增量处理”模式：

1. 采集任务定时运行，优先高价值栏目；
2. 数据写入后，依据 `crawled_at` 和 `id` 游标增量拉取；
3. 下游只处理新增数据，降低重复计算成本。

这一策略在资源受限环境下更稳定，也更符合毕业设计阶段持续演示需求。

### 3.9.2 数据治理与版本管理

随着规则迭代，建议对以下元素进行版本化：

- 清洗规则版本；
- `source_type` 分类规则版本；
- 采集策略版本（栏目配比与重试参数）。

版本化后可开展对比实验：例如规则 v1 与 v2 在“有效新闻率、抽取成功率、冲突检测质量”上的差异。

### 3.9.3 面向答辩的可解释证据链

从论文展示角度，第3章应能回答评委两个问题：

## 3.10 本章小结

本章围绕“数据如何进入系统并转化为结构化事实”完成了从策略到实现的全链路论证。首先，基于 OpenClaw Skill 明确了采集任务的目标、约束与验收标准，强调了栏目优先级和有效新闻过滤在数据价值控制中的核心作用；其次，结合 PostgreSQL `news` 表的 schema 与幂等入库策略，说明了如何在真实反爬约束下实现可持续的数据积累；再次，结合 `datasource` 模块的仓储层、适配层与服务层实现，展示了数据对象从数据库记录到 `PreprocessDocument` 的契约化迁移过程；最后，连接预处理与抽取链路，说明了第3章输出如何直接支撑第4章真实性验证和第5章检索问答。

与传统“仅介绍算法”的章节不同，本章强调了数据工程在知识系统中的基础性作用。对于动态新闻场景，模型能力并不能替代高质量数据入口。只有当采集策略、清洗规则、入库约束和对象契约形成稳定体系时，后续的知识图谱与问答服务才具备可扩展、可解释、可复现的工程价值。这也使第3章成为全文技术路线中最关键的一环。
