# 第2章 系统需求分析与总体设计

## 2.1 引言

在“动态足球新闻知识库”这一研究问题中，系统建设并非单一模型任务，而是一个从数据输入到知识服务的完整工程问题。与静态问答系统不同，本课题面向持续变化的新闻流：同一事件可能被多次报道、补充、修正甚至否定。如果系统仅追求“生成一段流畅回答”，而缺少来源可追踪、事实可更新、冲突可处理的机制，那么输出结果很难在学术与工程层面成立。

因此，本章的核心任务不是罗列功能清单，而是建立一套可执行的系统设计逻辑：先定义问题边界，再明确需求约束，随后给出分层架构、对象契约、部署方案与技术选型依据。后续章节将按此设计逐层实现：第3章讨论数据采集与入库，第4章讨论预处理与抽取，第5章讨论真实性验证与图谱构建，第6章讨论检索问答与系统服务化。

从论文整体结构看，第2章承担“设计基准章”角色。其质量直接决定后续章节是否具备统一叙事主线与工程闭环。本章将围绕以下问题展开：

1. 系统到底要解决什么问题，成功标准是什么；
2. 在真实约束下，功能与非功能需求如何分层表达；
3. 总体架构如何把采集、抽取、验证、问答串成可持续链路；
4. 为什么当前技术选型能够支撑毕业设计阶段的可落地目标。

## 2.2 问题定义与设计目标

### 2.2.1 研究对象与核心矛盾

本研究对象是英超新闻事实的动态知识构建与问答服务。系统面对的核心矛盾并非“是否能抽出实体”，而是“如何在多源、时变、存在冲突的新闻文本中维护可解释知识”。该矛盾具体表现为：

1. **来源异构**：官方公告、媒体报道、评论稿在语气与可靠性上差异显著；
2. **时间漂移**：同一事件在不同时间点可能有不同版本；
3. **语义粒度不一**：一篇新闻中常包含多个子事件与背景描述；
4. **工程链路长**：上游采集误差会沿链路放大到问答输出端。

这说明系统目标必须从“文本抽取”升级为“可信知识生产与服务”。

### 2.2.2 设计目标与成功标准

结合上述矛盾，系统总体目标定义为：构建从新闻采集到证据约束问答的端到端闭环，并保证知识对象可追踪、可更新、可解释。相应成功标准包括：

1. **链路完整性**：采集→预处理→抽取→验证→图谱→问答全流程可运行；
2. **对象一致性**：跨层字段语义稳定，关键标识可贯通；
3. **知识可信性**：支持来源加权、冲突识别、置信传播与版本更新；
4. **服务可用性**：可通过 API 进行检索问答，且回答受证据约束；
5. **实验可复现性**：固定样本可回放中间结果，支持论文论证。

换言之，本研究评价系统优劣，不仅看最终答案文本，更看“答案背后是否有完整证据链”。

### 2.2.3 设计原则

为保证目标可落地，系统遵循四项设计原则：

1. **契约先行**：先定义层间数据对象，再实现算法细节；
2. **分层解耦**：采集、抽取、验证、服务各层可独立演化；
3. **证据优先**：问答生成以前置检索和结构化事实为基础；
4. **可观测可回放**：中间结果可打印、可追踪、可复盘。

## 2.3 系统需求分析

### 2.3.1 功能需求

结合研究目标，系统功能需求可划分为七类：

1. **新闻数据接入与持久化**
   - 需支持从指定数据源持续获取新闻，并以结构化形式写入 PostgreSQL；
   - 必须具备去重机制和基本质量过滤，保证入库数据可用。

2. **文本预处理与语义组织**
   - 对长文本进行句级拆分和语义分块；
   - 保留来源与时间元数据，避免后续事实断链。

3. **事件抽取与锚点结构化**
   - 从语义块生成事件单元；
   - 提取参与者、约束、时间锚点、来源等结构化字段。

4. **多源真实性验证**
   - 依据来源先验和关系分析识别支持/冲突；
   - 输出初始置信、当前置信、状态与传播轨迹。

5. **知识图谱构建与版本管理**
   - 将事件、实体、来源、约束关系持久化到 Neo4j；
   - 支持旧事件置信更新与关系版本记录。

6. **GraphRAG 检索问答**
   - 支持自然语言查询解析、图检索、上下文构建与生成回答；
   - 回答应尽量引用可追踪事件证据，降低幻觉。

7. **系统服务与任务调度能力**
   - 提供 API 对外服务；
   - 支持异步任务框架承载长耗时处理。

### 2.3.2 非功能需求

非功能需求是系统可运行、可演示、可扩展的保障，本研究重点关注以下六项：

1. **可靠性**：模块异常不能轻易导致全链路崩溃，需支持局部失败隔离；
2. **可维护性**：代码结构与对象契约清晰，便于后续迭代；
3. **可扩展性**：可替换模型后端、扩展数据源与规则策略；
4. **可解释性**：关键判断过程（过滤、验证、检索）应有可复核依据；
5. **可部署性**：支持容器编排，降低环境差异带来的运行风险；
6. **可复现性**：固定输入样本可复现中间对象与关键输出。

### 2.3.3 约束条件与边界假设

毕业设计阶段存在现实约束：

1. 模型推理能力受本地/实验环境算力限制；
2. 数据源存在反爬与页面结构变化风险；
3. 工程周期有限，需优先保证主链路闭环而非全面产品化。

基于以上约束，系统采用“核心方法先闭环，平台能力后增强”的实施路线：先保证知识生产主链可运行，再逐步补齐监控、安全与治理能力。

## 2.4 总体架构设计

### 2.4.1 分层架构与职责划分

系统采用六层核心业务架构 + 一层服务编排架构：

1. **数据源层（datasource）**：数据获取、清洗入库、统一读取接口；
2. **预处理层（preprocess）**：分句、语义分块、文本单元组织；
3. **抽取层（extractor）**：事件分解与锚点抽取；
4. **验证层（verifier）**：支持/冲突分析与置信传播；
5. **图谱层（knowledge_graph）**：事件知识持久化与关系管理；
6. **问答层（rag）**：查询解析、图检索、上下文构建、受约束生成；
7. **服务编排层（api/workers）**：HTTP 服务与异步任务执行。

该分层对应“数据进入—知识生产—知识服务”三段式链路，具备明确职责边界，便于论文按模块展开技术论证。

### 2.4.2 主流程与控制流程

系统运行可拆分为两类流程：

1. **离线/批处理主流程**：
   数据采集 → 文本预处理 → 事件抽取 → 真实性验证 → 图谱写入；
2. **在线问答流程**：
   用户问题 → 查询解析 → 图谱检索 → 上下文构建 → 答案生成。

两类流程在知识图谱层汇合：离线流程负责“持续供给可信知识”，在线流程负责“消费可信知识并服务用户”。这种解耦可避免在线问答直接依赖高耗时抽取流程，提升系统响应稳定性。

### 2.4.3 架构权衡分析

本系统在设计中做了三项关键权衡：

1. **精度 vs 吞吐**：采用两阶段抽取（分解+锚点）牺牲部分实时性，换取结构稳定性；
2. **灵活性 vs 一致性**：允许模型能力扩展，但通过契约与后处理强制字段统一；
3. **研究深度 vs 工程完整度**：优先实现可信知识主链，再扩展鉴权、监控等平台能力。

这些权衡符合毕业设计“方法可验证 + 系统可运行”的总体目标。

## 2.5 数据对象契约与可追溯设计

### 2.5.1 核心对象链路

系统核心对象链路如下：

$$
ArticleDocument \rightarrow PreChunkInput[] \rightarrow SemanticChunkDocument[] \rightarrow EventUnit[] \rightarrow AnchoredEvent[] \rightarrow ValidatedEvent[]
$$

该链路体现“逐层结构化”思想：每一层只解决本层问题，不越权承担下游职责，从而减少跨层副作用。

### 2.5.2 字段契约稳定性

关键字段在链路中需要稳定贯穿，包括：

- 来源字段：`source_name`、`source_type`；
- 时间字段：`publish_date`；
- 标识字段：`doc_id`、`block_id`、`event_id`；
- 语义字段：`block_text`、`event_description`。

字段稳定性的工程价值在于：

1. 降低跨模块调试成本；
2. 支持端到端断言测试；
3. 为审计与可解释输出提供基础。

### 2.5.3 分层 ID 与追踪机制

系统采用分层 ID 规范：

- 文档：`news-{id}`
- 句子：`{doc_id}:s{序号}`
- 语义块：`{doc_id}:b{序号}`
- 事件：`{chunk_id}:e{序号}`

该规范实现了“问题可定位、结果可回溯”。当问答输出某事件时，可反向追踪到对应块文本和原始新闻 URL，满足论文中的证据闭环要求。

## 2.6 部署架构与运行模式

### 2.6.1 容器化服务拓扑

系统采用 Docker 容器化与 Compose 编排，核心服务包括：

1. `api`：提供接口入口；
2. `worker`：执行异步任务；
3. `postgres`：存储新闻原始数据；
4. `redis`：任务队列与结果后端；
5. `neo4j`：存储图谱知识。

该拓扑将“数据存储、任务调度、知识服务”拆分为独立组件，便于调试和迁移。

### 2.6.2 运行时协同机制

运行时分为两条通道：

- **同步通道**：API 直接响应查询类请求；
- **异步通道**：Worker 消费任务队列执行长耗时处理。

同步与异步分离能有效降低阻塞风险，尤其在批处理或重构图谱时可保护在线服务稳定性。

### 2.6.3 可复现部署价值

容器化不仅是工程实现手段，也是论文实验保障手段。统一镜像和配置可减少“环境差异导致结果不一致”的风险，提高答辩与复现实验可信度。

## 2.7 技术选型与合理性分析

### 2.7.1 基础框架与中间件

- **FastAPI**：接口定义简洁、文档化友好、异步支持良好；
- **Celery + Redis**：适配爬取与批处理等长任务；
- **PostgreSQL**：结构化数据管理与增量读取稳定；
- **Neo4j**：天然适合事件—实体—来源关系表达与图检索。

### 2.7.2 方法层技术选择

- **LLM 语义能力**：用于分块评分、事件分解、锚点抽取与查询解析；
- **规则/阈值机制**：用于失败回退、结构约束与质量门禁；
- **GraphRAG 架构**：先检索后生成，强化证据约束。

### 2.7.3 选型与目标匹配

本研究的技术栈并非追求“最新”，而是追求“与研究目标匹配”：既能支撑完整知识链路运行，又能在有限周期内完成验证与展示。通过“成熟组件 + 明确契约 + 可回放测试”组合，系统具备毕业设计所需的可实现性与可解释性。

## 2.8 本章小结

本章围绕“系统应该如何被设计”完成了从问题定义到总体架构的完整论证。首先，明确了动态新闻知识系统面临的核心矛盾与成功标准；其次，提出了覆盖功能与非功能的需求体系，并给出了现实约束下的实施边界；再次，构建了分层架构、对象契约、ID 追踪与部署拓扑，形成可执行的工程蓝图；最后，说明了当前技术选型与研究目标之间的匹配关系。

在章节衔接上，第2章提供的是“设计约束与方法框架”，第3章至第6章提供的是“分层实现与实证支撑”。二者共同构成了本论文从理论目标到工程落地的主线结构。
