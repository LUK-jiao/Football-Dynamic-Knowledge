# 第6章 检索问答与系统实现

## 6.1 引言

在前文完成事件抽取、真实性验证和知识图谱构建后，系统进入“知识服务化”阶段，即如何让用户通过自然语言直接访问图谱中的可信事实。该阶段的核心目标不是生成一段“看起来合理”的回答，而是构建一条可追溯、可解释、可扩展的检索问答链路。换言之，问答结果应尽可能由图谱证据驱动，并在表达层面保持可读性，在工程层面保持可部署性。

针对这一目标，本文采用 GraphRAG 思路实现问答引擎：首先将用户问题解析为结构化检索约束；然后在知识图谱中执行参数化查询召回候选事件；接着将事件集合组织为面向生成模型的上下文；最后在约束提示下生成答案。该流程本质上是“检索先行、生成后置”的证据约束范式，能够显著降低纯生成问答中的幻觉风险。

此外，系统在实现层采用 FastAPI 与 Celery/Redis 组合，形成“在线问答服务 + 异步任务框架”的双层工程架构。前者面向交互响应，后者面向耗时任务解耦。通过这一设计，系统不仅能够完成毕业设计所需的功能闭环，也为后续扩展留出了标准化接口与调度空间。

## 6.2 GraphRAG 检索引擎

GraphRAG 引擎由四个核心组件构成：查询分析器（QueryAnalyzer）、图检索器（GraphRetriever）、上下文构建器（ContextBuilder）与生成后端（RAGLLMBackend），并由统一编排器（GraphRAG）串联。系统标准处理流程为：

1. `parse(query)`：自然语言问题结构化；
2. `retrieve(parsed_query)`：参数化 Cypher 检索事件；
3. `build/context`：按意图组织上下文；
4. `chat(prompt)`：约束生成最终回答。

该流程的关键优势在于：每一步均有明确输入输出边界，便于独立调试和后续替换。例如，可在不改动检索层的情况下替换生成模型，也可在不改动模型层的情况下迭代查询解析策略。

### 6.2.1 查询分析器（QueryAnalyzer）

查询分析器承担“自然语言到结构化查询”的语义映射任务。其输出结构包含五类核心字段：

- `entities`：实体列表（对象形式，含 `name` 与可选 `entity_type`）；
- `constraint_types`：事件语义类别（九类约束锚点）；
- `time_filter`：时间过滤（`mode/start/end`）；
- `intent`：问题意图（`fact/summary/analysis`）；
- `limit`：结果上限（默认事实 20、总结/分析 50）。

在实现策略上，系统采用“LLM 主解析 + 规则回退”的双通道机制。主解析阶段使用低温参数（默认 `temperature=0.1`）要求模型输出严格 JSON。为提升鲁棒性，系统设置了两次 LLM 尝试；若仍失败，则进入规则解析。规则解析包括：

1. 关键词字典匹配约束类型（覆盖 MATCH_ACTION、MATCH_OUTCOME、PLAYER_MOVEMENT 等九类）；
2. 正则与时间规则识别（年份、具体日期、“最近”等表达）；
3. 基于词形特征的实体候选抽取；
4. 基于问题词判断意图（事实、总结、分析）。

这种设计体现了“语义能力与确定性能力互补”的工程思想。LLM 能处理复杂表达和隐含意图，规则层则在模型失效时提供最低可用保障，避免问答链路整体中断。对于本科毕业设计场景，这一方案在实现复杂度与系统稳定性之间取得了较好平衡。

### 6.2.2 图检索器（参数化 Cypher）

图检索器负责将结构化查询对象翻译为可执行 Cypher，并返回事件级证据集合。其核心做法是构建动态查询片段：

- 以 `MATCH (e:Event)` 作为主轴；
- 按需附加实体约束（`INVOLVES`）；
- 按需附加语义约束（`CONSTRAINS`）；
- 按需附加时间约束（`event_date` 区间）；
- 最后通过 `OPTIONAL MATCH` 回收实体、来源、约束、标题锚点。

与字符串拼接式查询相比，参数化 Cypher 具有两点优势：第一，查询结构与参数值分离，避免注入风险并提升可维护性；第二，统一模板可复用，便于后续缓存、监控与性能调优。

在检索策略上，系统区分“基础检索”和“分析扩展”两种模式：

1. **基础检索**：用于 `fact` 和 `summary` 问题，按约束直接返回事件集合；
2. **分析扩展**：用于 `analysis` 问题，先完成基础检索，再基于标题锚点扩展相关事件子图。

分析扩展的实际意义是把离散事件提升为“事件簇”或“事件链”，为因果解释、趋势分析和阶段总结提供更完整证据上下文。最终返回的每条事件记录通常包含：`event_id`、`event_description`、`event_date`、`fact_type`、`entities`、`sources`、`constraints`、`titles`。

### 6.2.3 上下文构建器

上下文构建器是检索层与生成层之间的关键桥梁。若直接把原始检索结果输入模型，常见问题是信息重复、时序混乱、上下文过长。为此，系统采用“去重—排序—裁剪—格式化”的四步构建策略：

1. 按 `event_id` 去重；
2. 按时间排序（默认近到远，分析模式为时间正序）；
3. 按 `max_events` 截断（默认 30）；
4. 转换为模板化文本块。

为了适配不同任务目标，系统设计了三套上下文模板：

- **事实模板（build）**：强调单事件要素完整，便于精准问答；
- **总结模板（build_summary_context）**：按约束类型聚合，便于横向概览；
- **分析模板（build_analysis_context）**：按时间线展开，便于推导演化过程。

在生成阶段，系统还会根据意图选择不同提示词模板：事实问答强调“仅依据上下文、引用事件ID、拒绝臆造”；总结问答强调“聚合与统计”；分析问答强调“时序关系、原因与影响解释”。这种“意图驱动的上下文与提示协同”是本系统问答质量提升的关键点之一。

## 6.3 系统 API 与异步任务（api/workers）

在工程实现层，系统采用 FastAPI 作为服务框架，并使用 Celery + Redis 构建异步任务基础设施，形成清晰的分层结构。

### 6.3.1 API 服务组织

应用入口统一定义了服务元信息、跨域策略、健康检查和生命周期钩子，并按业务域注册路由，包括数据源、知识管理、检索、RAG 与反馈模块。此种模块化路由组织能够降低耦合度，便于后续分工开发和版本迭代。

在依赖管理方面，系统已预置配置注入、数据库会话依赖与用户身份依赖等机制。虽然当前鉴权与数据库会话仍处于原型占位阶段，但接口层已经具备“可替换依赖”的结构前提，可在后续快速接入 JWT 校验、真实会话管理与权限控制。

RAG 路由当前提供标准化请求/响应契约：输入查询文本，输出答案、来源与置信信息。该“接口先行”策略保证了系统外部协议稳定，即使内部编排仍在迭代，也不会频繁破坏前后端联调与测试流程。

### 6.3.2 异步任务框架

针对爬取、批处理和向量更新等长耗时任务，系统通过 Celery 建立后台执行通道。当前配置已包含：

- Broker 与 Result Backend（Redis）；
- JSON 序列化；
- UTC 时区；
- 任务状态跟踪；
- 软硬超时控制（25/30 分钟）。

已定义的任务类型包括：网站爬取、文档处理、向量更新。虽然任务逻辑仍为占位实现，但任务协议、队列入口和执行配置已经完成，这意味着系统具备从“同步串行处理”向“异步并行处理”迁移的基础能力。

### 6.3.3 当前实现程度与工程意义

从本科毕业设计角度看，系统实现呈现出“核心方法可运行、平台能力可扩展”的阶段特征：

1. GraphRAG 主链（解析—检索—上下文—生成）已形成可运行闭环；
2. API 与任务系统已完成骨架化与协议化；
3. 安全鉴权、任务业务化与服务化监控仍需在后续工程阶段补齐。

这种分层推进策略符合毕业设计常见路径：先验证方法可行性与系统闭环，再逐步补强工程能力。

## 6.4 本章小结

本章围绕“可信知识如何转化为可用问答能力”展开，给出了 GraphRAG 引擎与系统实现方案。方法层面，系统通过查询分析、参数化图检索和上下文构建，将自然语言问题转换为证据约束下的生成任务，降低了幻觉并提升了答案可解释性。实现层面，系统基于 FastAPI 与 Celery 构建了服务与任务双通道架构，形成了清晰的模块边界与扩展路径。

结合第5章可见，本研究已形成从“事实抽取与验证”到“检索问答服务化”的完整技术链路：前者保证知识质量，后者保证知识可用。该链路不仅满足毕业设计对系统完整性的要求，也为后续性能优化、权限治理和业务部署提供了可持续演进基础。
